# semver-leap

> A LEAP port of [node-semver](https://github.com/npm/node-semver) — the semantic version parser used by npm.
>
> **Generated entirely from prompts, schemas, and tests. Zero hand-written application code.**

## What is this?

An experiment to answer a question:

> **What if code is a commodity and the guardrails are the product?**

Instead of writing code, this project contains:

- **Prompts** — declarative intent (`master.md` at root + one per part)
- **Schemas** — contracts between components (`schemas/`)
- **Tests** — behavioral guardrails, ported verbatim from node-semver (`tests/`)

The actual implementation in `src/` is **gitignored**. It doesn't exist until an AI agent generates it from the prompts.

```bash
# Zero code in the repo. Run this to produce a working semver parser:
git clone https://github.com/safitudo/semver-leap.git
cd semver-leap
npm install

# Then, in Claude Code:
claude "build the app, start from master.md, run the tests"
```

## The LEAP Framework

This is a proof-of-concept for **LEAP — LLM Engineered Application Pattern**. See [leapfrog-skill](https://github.com/safitudo/leapfrog-skill) for the installable Claude Code plugin.

Three artifacts, ranked by value:

1. **Tests** — human-authored, behavioral. The definition of "correct."
2. **Schemas** — the architecture. Contracts between parts.
3. **Prompts** — the intent. Disposable — any prompt that passes the tests is valid.

Everything else is generated.

## Analysis: LEAP vs the original

After generating `src/` and running the full upstream test suite:

### Where LEAP won

- **5,632 / 5,632 test assertions passing** — faithful behavior including ReDoS guards, whitespace edge cases, prerelease ordering, subset semantics, right-to-left coerce.
- **File structure parity** — 48 vs 45 files, identical component boundaries. The `parts/` decomposition gave the agent crisp targets.
- **~10% less code** — generated files are consistently shorter than upstream.
- **More idiomatic in places** — e.g., `compare()` uses short-circuit `||` (3 lines) vs the original's explicit if-blocks (19 lines).
- **~3.5× leverage** — 717 lines of spec produced 2,540 lines of working implementation.

### Where LEAP lost

- **Lost institutional knowledge** — the `subset.js` spec comment shrank from 30 lines to 4. Fifteen years of "why we do it this way" comments evaporated. This is the biggest real cost.
- **Dropped `.js` extensions in requires** — subtle regression that could break in strict ESM environments.
- **Slightly different error wording** — could break downstream code asserting on error strings.
- **No fuzz testing beyond vendored tests** — subtle behavioral deltas could lurk.

### Caveats

`node-semver` was an unusually favorable target:

- Small (~2.5k LOC)
- Public BNF grammar at [semver.org](https://semver.org)
- Gold-standard test suite usable verbatim as the oracle
- Algorithms documented publicly
- Pure functions, no I/O

A greenfield library without those advantages would be a harder test. But — `node-semver` is also **a critical dependency for every JavaScript project on Earth**, which is why it's a meaningful proof point.

## What this suggests

Most open source libraries aren't novel algorithms. They're careful handling of edge cases discovered over time. Those edge cases are (or should be) in the test suite. **If the edge cases are in tests, LEAP can eat them.**

The 15 years of `node-semver`'s commit history is mostly a long trail of "fix edge case X, add test for Y." That accumulated wisdom lives in the test suite now — and the test suite is what LEAP treats as the source of truth.

## Project structure

```
semver-leap/
├── master.md              # Root prompt — "read this first"
├── parts/                 # Per-component prompts
│   ├── classes/           # SemVer, Range, Comparator
│   ├── functions/         # compare, satisfies, valid, etc.
│   ├── ranges/            # subset, intersects, simplify, etc.
│   ├── internal/          # regex bank, identifiers, lru-cache
│   └── bin/               # CLI
├── schemas/               # Shared TypeScript contracts
├── tests/                 # Behavioral tests (ported from node-semver)
├── tap-snapshots/         # Snapshot fixtures for CLI tests
└── src/                   # ★ GITIGNORED — generated by AI
```

## Running the tests

```bash
npm install
# After an AI agent generates src/:
npm test
```
